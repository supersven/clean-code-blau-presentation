Clean Code @ blau

Eine kurze Vorstellung

Sven Tennie (sven.tennie@blau.de)

Sauberer Code ...

  • kann gut gelesen und leicht verstanden werden
  • kann einfach erweitert werden
  • ist offensichtlich korrekt (keine versteckten Fehler)
  • ist gut getestet
  • dient als Dokumentation für die Anforderung / das Feature

Warum?

  • Code wird sehr viel häufiger gelesen als geschrieben
  • 10 Jahre blau - so lange bleibt kein Entwickler in einer Firma... und kein Anforderungsdokument aktuell
  • Stabilität & Korrektheit
    • Fehler in unserer Software könnten richtig teuer werden
    • die Admins wollen nachts lieber schlafen, als unsere Fehler zu behandeln

Warum?

  • Erweiterbarkeit
    • blau passt sich möglichst schnell an Rechtsprechung und Markt an
    • der Aufwand für eine Erweiterung muss (einigermaßen) konstant bleiben, sonst bekommt man mittelfristig ein Problem

Wie?

Test Driven Development

  • wir stecken viel Arbeit in die Tests
    • Test- und Produktions-Code werden gleichwertig behandelt
    • Tests werden stets refactored und gepflegt
  • möglichst "Test First"
  • wir streben nach 100% Testabdeckung
  • wir vertrauen unseren Tests
    • wenn alle Tests grün sind, releasen wir

Refactoring

  • wenn wir etwas verbessern können, tun wirs
  • immer Zeit fürs Aufräumen und Verbessern eingeplant
  • die Tests stellen sicher, dass wir dabei nichts kaputt machen
  • der Code gehört uns allen gemeinsam
    • jeder darf alles verbessern

Reviews

  • je nach Team und Bedarf machen wir Code Reviews
  • am Beamer mit dem ganzen Team
  • konstruktive Diskussionskultur
  • Wissenstransfer
  • Code muss für jeden gut verständlich sein

Pair Programming

  • wir arbeiten immer eng zusammen
  • häufig zu zweit an einem Rechner
    • gemeinsames Verständnis des Problems
    • Wissenstransfer
    • weniger Fehler

Regeln

  • Clean Code von Uncle Bob

Regeln

  • persönliche Erfahrungen
  • nach einigen Pairing- und Review-Runden hat jeder die wichtigsten Regeln im Kopf
  • Diskussionsergebnisse aus Reviews
  • kein starres Regelwerk - aber man braucht fürs Review einen guten Grund es anders zu machen ... ;)
